Typical Discovery Timeline
A real discovery usually spans 2-3 weeks across multiple client interactions. The system continuously tracks progress and guides the PO toward readiness.
1
First Client Meeting
PO uploads meeting notes. System ingests → extracts facts → evaluates readiness at ~25%. Generates 12 prioritized follow-up questions.
Intake Analysis Gap Detection
2
Email Exchanges + Second Meeting
Client sends docs (API specs, existing platform). PO uploads. Asks "prepare my meeting agenda" → system generates gap-driven agenda. Post-meeting: readiness rises to ~58%.
Intake Meeting Prep Control Point
3
Closing Gaps & Contradictions
PO sends targeted questions via email using gap analysis. Client responds. System detects contradiction ("Meeting 1 said single-tenant, email said multi-tenant"). PO resolves in next call.
Gap Detection Analysis Contradiction Alert
4
Repo Analysis (if available)
PO provides repo URL → Claude Code analyzes codebase → 12 technical facts extracted → readiness jumps. System suggests questions about internal APIs.
Claude Code Analysis Control Point
5
Document Generation & Handoff
Readiness reaches 87%. PO triggers "Generate discovery documents". System produces Discovery Brief, MVP Scope Freeze, Functional Requirements. PO reviews, minor edits, hands off to Phase 2.
Role Simulation Doc Generator Handoff → Phase 2

Why This Architecture
🎯
Deterministic, Not Probabilistic
Traditional RAG: "search and hope the answer is in some chunk." This system: control point checks against discrete, confirmed facts in Mem0. Readiness is measured, not guessed.
🔗
Traceable to Source
Every fact traces back to the meeting, email, or repo that produced it. Phase 2 team can always ask "where did this come from?" and get a definitive answer.
🔄
Stateless Agent Communication
Agents never talk to each other directly. They read and write through knowledge layers. This means any agent can be replaced, scaled, or retried without breaking the system.
