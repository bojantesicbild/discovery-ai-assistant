Discovery Phase — Data Flow

How data moves through discovery.html: from static stores → user interactions → DOM rendering

Layer 1 — In-Memory Data Stores

All data is hardcoded as JS objects. No API calls, no localStorage. State is ephemeral — lost on page reload.

Layer 2 — Viewer Functions (Data → DOM)

Each viewer reads from a data store, builds an HTML string, and injects it into an overlay panel.

⬡
openCpViewer(cpKey)
Reads cpData[cpKey]
→ SVG ring (pct + level)
→ breakdown bar (confirmed / assumed / gaps)
→ fact list (sorted, 5 visible + toggle)
→ related docs + open gaps
→ "Ask about this CP" button
👤
openPersonViewer(key)
Reads personData[key]
→ avatar + name/role
→ stats grid
→ meetings + decisions + activity
→ "@mention" button → pre-fills chat
📄
openDocViewer(docType)
Reads docContents[docType]
→ icon + title + meta
→ body (rich HTML)
→ footer references
✓
openReqViewer(reqId)
Reads reqDetails[reqId]
→ requirement detail + impact
→ Approve / Reject buttons
▼ ▼ ▼ ▼
innerHTML injection into overlay panels → overlay.classList.add('open')

Layer 4 — User Interactions (Events)

Actions triggered from within rendered panels or the main UI that mutate data or DOM state.

●
toggleFactStatus()
Click fact dot → toggle confirmed ↔ assumed
Mutates: cpData[cpKey].facts[i].status
Updates: dotEl.className
Persists: within session only
▸
toggleFacts()
Show/hide facts beyond the first 5
Toggles: display:none on [data-facts-extra]
Updates: button text
✓
approveReq() / rejectReq()
Approve or reject a requirement card
Adds: .approved / .rejected class
Updates: actions area HTML
@
@Mention / Ask / Gap
Pre-fills chat textarea from context
Sources: CP ask btn, person mention, gap ask
Target: textarea.value + focus
Cross-Cutting — Tab Switching, Keyboard, Workflows
⇥

⎋
ESC key handler
Close any open overlay or popup
cpOverlay → personOverlay → docOverlay → wfPopup
/
Workflow popup

Key Notes
No persistence
All state is in-memory JS objects. Refreshing the page resets everything. No localStorage, no API calls.
One-way render
Viewers do a full HTML rebuild from data on each open. No reactive binding — changes to data don't auto-update the DOM.
Fact toggle
Only mutation that writes back to a data store. Changing fact status updates both cpData and the dot's class, so re-opening preserves it.
Chat pre-fill
Multiple actions converge on the textarea: CP ask button, @mention, gap ask, workflow select. All write to textarea.value then focus.
