# Output Documents & Downstream Integration

## What Discovery Produces

The Discovery AI Assistant generates 6 document types. Three are the
primary handoff to Phase 2 (Story/Tech Doc Assistant). Three are working
documents used during discovery.

```
Primary Handoff (Phase 2 input):
├── Project Discovery Brief     — the big picture
├── MVP Scope Freeze            — what gets built
└── Functional Requirements     — feature-level detail

Working Documents (used during discovery):
├── Meeting Summary             — after each client interaction
├── Gap Analysis Report         — what's missing, what to ask
└── Multi-Perspective Analysis  — requirements from different roles
```

### Quality Bar for Handoff

Discovery documents are ready when:
- Phase 2 user can understand the project **without asking the PO basic questions**
- Assumptions are clearly labeled (so Phase 2 assistant can flag them)
- Scope is explicit — what's in, what's out, no grey areas
- Control points checklist is substantially complete (85%+ ready)

---

## How Documents Are Generated

The Document Generator Agent composes documents by pulling from all
knowledge layers:

| Content type | Source | Example |
|-------------|--------|---------|
| Narrative paragraphs | **RAGFlow** (document search) | "The client described their auth flow as..." |
| Structured data tables | **Mem0** (fact store) | Feature list with priorities and status |
| Stakeholder lists | **Mem0** (entity graph) | Key contacts with roles and decision authority |
| Technical context | **Claude Code** (repo analysis) | "Existing stack: React 18, Express, PostgreSQL" |
| Assumption markers | **Mem0** (fact store) | Facts with status = "assumed" flagged inline |
| Status indicators | **Control Point Agent** | Readiness score, area breakdown |

Every statement in a generated document is traceable to a source (which
meeting, which email, which repo analysis). Phase 2 users can verify
anything by checking the source reference.

## Downstream Integration

### What Phase 2 Receives

The Story/Tech Doc Assistant receives the three primary documents and uses
them to generate detailed user stories, PBIs, technical specs, and
architecture documents.

```
Discovery AI Assistant outputs:
├── Project Discovery Brief     ─┐
├── MVP Scope Freeze            ─┼──→  Story/Tech Doc Assistant (Phase 2)
└── Functional Requirements     ─┘     │
                                        ├── User stories
                                        ├── Technical specs
                                        ├── Architecture docs
                                        ├── decision.md  ──→ back to knowledge base
                                        └── learning.md  ──→ back to knowledge base
```

### Knowledge Feedback

Phases 2, 3, and 4 produce decision and learning markdown files as they
work. These documents can be ingested back into the Discovery AI Assistant's
knowledge base:

- **Decision logs** — "We chose PostgreSQL over MongoDB because..."
  → Stored as cross-project facts for future discoveries
- **Learning docs** — "Auth migration from JWT to SSO took 3 sprints"
  → Available when estimating similar work in future projects
- **Architecture decisions** — "Microservice split caused deployment complexity"
  → Surfaced in future discovery when similar patterns emerge

This feedback loop means every project makes the system smarter for the
next one.
