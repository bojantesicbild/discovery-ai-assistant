Contradiction Detection & Resolution
Meeting 3 decided "Microsoft SSO only for MVP". Two weeks later, Meeting 7 introduces "Social login for external learners". Here's exactly how the system detects, tracks, and helps resolve this contradiction.
A Detection — When the Contradiction Surfaces
Source A — Meeting 3 (Feb 10)
"Microsoft SSO as primary auth for MVP. All users authenticate through Azure AD. No other auth methods in MVP scope."
Decided by: David Miller (CTO) & Sarah Chen (Tech Lead)
Status: Confirmed — stored as fact since Feb 10
Source B — Meeting 7 (Yesterday)
"We need to think about external learners who don't have Microsoft accounts. Can we add Google and Apple login?"
Raised by: David Miller (CTO), supported by Lisa Wang (PM)
Status: Assumed — flagged as contradicting Source A
How the System Detected This
During Mem0 fact extraction, the new fact "Social login (Google, Apple) requested for external learners" was compared against all existing auth-related facts.

Match found: Existing fact "Microsoft SSO only for MVP — no other auth methods" (confirmed, Meeting 3).
Semantic similarity: 0.89 — both discuss MVP authentication scope.
Contradiction signal: "no other auth methods" vs. "add Google and Apple login" — direct conflict.

The Analysis Agent also checks the entity graph: both facts are attributed to David Miller, strengthening the contradiction flag (same stakeholder, changed position).
B Tracking — What the System Records

ID: CONTRA-012

Severity: High — affects architecture, timeline, and scope

Position A: Microsoft SSO only (Meeting 3, Feb 10) — David Miller, Sarah Chen

Impact if Position B adopted:
