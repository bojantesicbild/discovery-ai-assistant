Knowledge Architecture
The system uses 4 specialized knowledge layers. Each serves a different purpose — and the key insight is that traditional RAG (Layer 1) alone cannot power deterministic readiness checks. That requires Layer 2 (facts). Here's how they work together.
Layer 1
Document Search
RAGFlow — Elasticsearch + MinIO
Stores full document content as chunks
Hybrid search: vector (0.7) + keyword (0.3)
RAPTOR parent-child chunking
Auto-keywords & auto-questions per chunk
13+ reranking providers
Best for: "What did the client say about X?"
Layer 2
Fact Store
Mem0 — PGVector + Neo4j
Discrete, deduplicated knowledge units
LLM-powered: ADD / UPDATE / DELETE / IGNORE
Version history with timestamps & sources
Confirmed vs. Assumed status per fact
Entity deduplication ("Sarah Chen" = "S. Chen")
Best for: "Do we know X? Yes/No."
Layer 3
Entity Graph
Neo4j
People, Orgs, Features, Decisions
Integrations, Constraints, Dependencies
Relationships: decided, requires, blocks
Stakeholder mapping & decision provenance
Best for: "Who decided X? What depends on Y?"
Layer 4
Code Understanding
Claude Code — Claude API
Architecture & tech stack analysis
API surface & integration points
Technical debt & code quality
Outputs feed into Layers 1 & 2
Best for: "What does the existing code do?"
Same Question, Different Layers

The PO asks: "What authentication method are we using?" — watch how each layer answers differently.

Advantage: Shows who decided, what depends on what, and the full decision chain. Enables: "Why MSAL.js?" → "Because Sarah Chen chose it in Meeting 7, and it's supported by the client's existing Microsoft 365 infrastructure."
Layer 4 — Claude Code (Repo Analysis)
"What authentication method are we using?"
Returns what the existing code actually does:

• No SSO integration present

Advantage: Grounds technical decisions in reality. Sarah chose MSAL.js, and Raj confirmed the gateway can handle it — Claude Code confirms the gateway's architecture is compatible. No surprises in Phase 2.
The Critical Difference
Layer 1 alone (traditional RAG): "Here are some relevant paragraphs — good luck figuring out the current state."
Layer 1 + Layer 2: "Auth method is MSAL.js, confirmed. Social login was considered but deferred. Control point: covered."
All 4 layers: "MSAL.js confirmed by Sarah Chen, depends on Azure AD, existing code supports it, social login deferred by David Miller in Meeting 8. Readiness for this item: 100%."

This is why control point readiness is deterministic, not probabilistic. We don't search and hope — we check discrete facts.

Role Simulation in Action
Before generating final documents, the Role Simulation Agent analyzes requirements from 5 cognitive perspectives. Here's REQ-045 (Social Login) examined through each lens — revealing conflicts that a single viewpoint would miss.

Based on Tarik Zaimovic's cognitive simulation methodology.
Requirement Under Analysis
REQ-045: Social login support (Google, Apple)
Priority: Should Have  |  Status: Assumed  |  Source: David Miller, Meeting 7
"Users should be able to sign in using their Google or Apple accounts, in addition to Microsoft SSO."
Perspective 1
🧑 End User
Reduces friction — external learners don't need a Microsoft account they'll never use elsewhere
Familiar flow — "Sign in with Google" is a pattern users already trust
Account confusion — if a user signs in with Google once and Microsoft later, are they the same person?
Mobile experience — social login is significantly smoother on mobile (no password typing)
Verdict: Positive — improves accessibility for external learners
Perspective 2
⚙️ Platform Admin
Identity management complexity — now managing 3 identity providers instead of 1
Account linking — need admin tools to merge/unlink accounts when users sign in with different providers
Security policy gaps — can MFA be enforced on Google/Apple accounts? Not natively through Azure AD
Audit trail — need unified logging across all auth providers
Verdict: Cautious — significantly increases admin overhead
Perspective 3
💻 Developer
+2 sprints estimated — OAuth 2.0 integration with Google + Apple, account linking logic, session unification
Apple Sign In quirks — email relay, limited user info, different token format than Google
Token management — refresh tokens from 3 providers with different expiry policies
MSAL.js reusable patterns — OAuth flow architecture can be extended for social providers
Verdict: High effort — viable but non-trivial, not recommended for MVP
Perspective 4
📊 Business Owner
Market reach — external learners are a growth segment; frictionless onboarding drives adoption
Timeline risk — +2 sprints threatens Q3 2026 launch target
ROI unclear — what % of users would actually be external learners at launch?
Competitive advantage — competitors require enterprise SSO only; social login differentiates
Verdict: Strategically valuable but premature for MVP
Perspective 5
🧪 QA Engineer
Test matrix explosion — 3 auth providers × account states × session scenarios = 3x testing effort
Account linking edge cases — same email, different provider; provider account deleted; email changed
Provider outage testing — what happens when Google/Apple auth is down? Graceful fallback needed
Security testing — CSRF, token theft, session hijacking across 3 providers
Verdict: Significantly increases test scope and security risk surface
Conflicts Between Perspectives
⚡
End User vs. Developer
Users want frictionless login; developers estimate +2 sprints and significant complexity. The user benefit is real but the implementation cost is high for MVP.
Resolution: Defer to Phase 2. Ship with SSO — add social login when the auth architecture is stable and the team has bandwidth.
⚡
Business Owner vs. QA
Business sees competitive advantage and growth; QA sees 3x test matrix and security risk. Both are right — the question is timing.
Resolution: Validate external learner demand before investing. If <10% of launch users are external, social login can wait.
Synthesized Recommendation
REQ-045: Defer to Phase 2

• 4 of 5 perspectives recommend deferral (End User is the exception, and even they flag account confusion risk)
• Developer: +2 sprints, non-trivial complexity
• QA: 3x test scope increase
• Business: timeline risk outweighs near-term value
• Admin: identity management overhead significant

Proposed path: Ship MVP with Microsoft SSO. Track external learner sign-up requests as a metric. If demand validates, prioritize social login for first post-launch sprint.

This recommendation aligns with: David Miller's Meeting 8 decision to defer social login.

Dimension
Traditional Discovery
Manual, PO-dependent
AI-Assisted Discovery

Meeting Input PO takes notes during meeting, types them up afterwards. Key details often lost or paraphrased. Depends on PO's note-taking skill. Google Meet recording auto-pushed to system. Whisper transcribes with speaker diarization. Every word captured, attributed to who said it.
Knowledge Tracking Confluence pages, spreadsheets, PO's memory. Facts scattered across 10+ documents. No single source of truth. Mem0 fact store — every fact is discrete, sourced, timestamped, and deduplicated. Confirmed vs. assumed. One source of truth.

When Phase 2 Finds a Problem

Discovery documents are not perfect. Sometimes Phase 2 discovers an assumption was wrong, a requirement is ambiguous, or a constraint was missed. Here's the feedback loop.

Example: Tech Lead Finds Ambiguity
Scenario: Tech Lead reads REQ-047 ("Session timeout configurable per role") and asks: "Per-role or per-user? The requirement says per-role but the meeting notes mention per-user flexibility."

What happens:
1. Tech Lead clicks the source link → sees Meeting 7 transcript: Raj said "per-role" but also mentioned "more flexibility"
2. The assumption is flagged — system creates a gap: "Session timeout granularity: per-role vs. per-user?"
3. Gap is routed back to PO → PO confirms with Raj → answer: "per-role for MVP, per-user in Phase 2"
4. Fact updated in Mem0, requirement clarified, decision logged with provenance

Key: The source traceability meant the Tech Lead could verify the original context in seconds, not days. The feedback updated the knowledge base for future reference.

Control Point Evaluation
How the system evaluates each checklist item against RAG data, assigns confidence scores with evidence, and calculates the weighted readiness score. This is the engine behind the 89% number — not a guess, but a deterministic, auditable evaluation.

How a Single Item Is Evaluated

The CP Agent uses an LLM call (configurable: Haiku for speed, Sonnet for accuracy, Opus for critical items) with a structured prompt. Here's the anatomy of one evaluation.

Example — Checklist Item: "Auth method confirmed"
Step 1: Query Mem0
GET /facts?category=authentication&status=confirmed
Returns: "Microsoft SSO via MSAL.js — confirmed, Meeting 7, Sarah Chen"

Step 2: Query Neo4j
MATCH (d:Decision)-[:ABOUT]->(t:Technology {name:"MSAL.js"}) RETURN d
Returns: Decision node with provenance chain → Sarah Chen → Meeting 7 → rationale

Step 3: LLM Evaluation
Prompt includes: checklist item definition, relevant facts, graph context, evidence requirement level

Step 4: Structured Output
{
  "item": "auth_method_confirmed",
  "status": "complete",
  "confidence": 0.98,
  "evidence": [
    { "type": "fact", "id": "f-047", "text": "Microsoft SSO via MSAL.js", "source": "Meeting 7" },
    { "type": "graph", "id": "d-012", "text": "MSAL.js decided by Sarah Chen", "source": "Neo4j" }
  ],
  "reasoning": "Auth method explicitly confirmed by Tech Lead with specific technology choice and rationale."
}
Item Status Classification

Each checklist item is classified into one of three statuses based on the evidence found.

Complete
Criteria: ≥1 confirmed fact directly addresses the item. Confidence ≥ 0.8.

Example: "Auth method confirmed" → Fact F-047 confirms MSAL.js, decided by Sarah Chen in Meeting 7.

Score contribution: Full weight applied (e.g., 8% of total).
Partial
Criteria: Related facts exist but are assumed (not confirmed), or evidence is indirect/incomplete. Confidence 0.4–0.79.

Example: "Offline mode defined" → Fact F-061 mentions "mobile-first" but offline capability is assumed, not confirmed.

Score contribution: Half weight applied (e.g., 2.5% of 5%).
Missing
Criteria: No relevant facts found. Confidence < 0.4. The topic has not been discussed or documented.

Example: "Offline mode defined" → No facts about offline capability exist. Zero evidence.

Score contribution: Zero weight. This becomes a gap for Meeting Prep Agent.
Weighted Scoring Formula

Not all checklist items are equal. A missing auth strategy is more critical than a missing browser support matrix. Weights are configurable per project template in Settings.

Formula
Score = Σ (item_weight × status_multiplier) / Σ (item_weight) × 100
where status_multiplier: complete = 1.0, partial = 0.5, missing = 0.0

Confidence Scoring

Confidence reflects how strong the evidence is, not just whether it exists. Two "complete" items can have different confidence levels.

High Confidence (≥ 90%)
"Auth method confirmed" — 98%

• Confirmed fact (not assumed)
• Named person decided it (Sarah Chen)
• Specific meeting reference (M7)
• Graph provenance chain exists
• No contradicting facts

Multiple independent evidence sources pointing to the same conclusion = high confidence.
Medium Confidence (50–89%)
"Deployment strategy outlined" — 55%

• Azure hosting mentioned (1 fact)
• No CI/CD details discussed
• No staging environment plan
• No deployment frequency target
• Evidence is thin — topic was touched, not covered

Some evidence exists but gaps remain. The item is "started" not "done".
Score History & Regression Detection

The system tracks readiness score after every ingestion event. This enables three critical alerts.

Regression Alert
Score drops after new data. Example: Meeting 7 introduces social login contradiction → Integration Points drops from 78% to 72%.

Trigger: Score decreases by ≥ 3% in any category.
Action: Flag contradiction, block affected document generation.
Stall Alert
Score plateaus despite new meetings. Example: 3 meetings happen but NFR score stays at 35%.

Trigger: Category score unchanged after 2+ ingestion events.
Action: Meeting Prep Agent prioritizes this category in next agenda.
Scope Alert
New requirements add checklist items that weren't in the original template.

Trigger: New requirement touches uncovered domain (e.g., mobile-specific items).
Action: PO notified, new items added to checklist, weights redistributed.

Low ≥ 1 related fact (any status) Assumed facts, indirect mentions, inferred context Early discovery, exploration phase, internal projects
Medium ≥ 1 confirmed fact + source attribution Confirmed facts with meeting/document source Standard projects, client work, most use cases
High ≥ 2 confirmed facts + graph provenance Multiple confirmed sources, decision provenance chain in Neo4j Regulated industries, large-scale projects, high-risk domains
Why This Matters
Traditional readiness is a snapshot opinion — one person's gut feeling at one moment in time. This system provides:

• Auditability — every score traces back to specific facts, meetings, and people
• Objectivity — same data = same score, regardless of who runs it
• History — score progression reveals patterns (stalls, regressions, breakthroughs)
• Actionability — missing/partial items become the next meeting's agenda
• Configurability — weights, thresholds, and evidence levels adapt to project type and risk profile

The 89% isn't a number someone picked. It's 14 complete items × weight + 3 partial items × half weight + 1 missing item × zero, with every claim backed by traced evidence.
